Problem

Interface (4 properties × 29 entries) is nearly as wide as the data it describes — shallow. `detected` is always coupled to `enabled`. skills_path is derivable from config_path. Every entry repeats the same 4-field pattern with no computed behaviour.

Solution

Replace the dict of 29 tool objects with an array of ToolConfig {name, path, enabled}. Derive skills_path, config_path, and detected as computed fields. The sync script reads the same seam — no changes needed at call sites.

  • Leverage: one interface change (add a computed field) propagates to 29 entries automatically
  • Locality: path convention lives in one place — the reader/resolver — not repeated 29 times
  • Interface shrinks: 4 props → 3 per entry; ~100 lines → ~35 lines
  • Test surface: a single small test validates the resolution; old config readers test themselves

Problem

Three scripts re-implement the same ROOT/CONFIG resolution, .env loading, and ${VAR} substitution independently. No shared module — each script's interface (flags) is simple but the implementation repeats ~26 lines. A bug in the substitution logic must be fixed in two places.

Solution

Extract a scripts/lib/common.sh module providing resolve_root(), load_env(), substitute_vars(), and a composite resolve_config() that returns a validated JSON dict. Each script sources the shared module and keeps only its unique logic.

  • Leverage: add a flag to resolve_config() and every script gets it
  • Interface shrinks: each script drops ~8 lines; one shared module at ~15 lines
  • Test surface: a single test file validates common.sh across all consumers

Delete the Orphan Category System

skills/apple/ · skills/mlops/ · skills/creative/ … + 17 more

Strong in-process

Files

  • 80 symlinks, 20 DESCRIPTION.md files
  • 3 empty dirs: file-organization/, last30days-skill/, write-a-skill/

Problem

The category system (DESCRIPTION.md + symlinks) has zero consumers. The sync script flattens by realpath. No tool reads DESCRIPTION.md. No tool groups skills by category. The deletion test is decisive: removing it all makes no behaviour vanish — it was a pass-through. The interface (symlink + DESCRIPTION.md format) was never used.

Solution

Delete all category directories, their symlinks, and DESCRIPTION.md files. Replace with metadata/tags in each SKILL.md frontmatter for any future grouping need. Category information becomes data (in the skill itself) rather than structure (a directory hierarchy that nothing reads).

  • Locality: each skill's category metadata lives inside its own SKILL.md
  • Leverage: a tag-based query replaces browsing nested directories — more powerful, less structure
  • Delete ~100 files (20 DESCRIPTION.md + 80 symlinks + 3 empty dirs)
  • Removes confusion: no more "why does mlops/ have subdirs with SKILL.md but apple/ has symlinks?"

Problem

The marketplace sub-system has half an interface — source configs in config.json and cache files on disk — but no implementation connects them. The cache is 440KB of orphan data. There is no install command, no provenance writer, no fetch refresh. 144/145 skills lack meta.json because no tool generates it.

Solution

  • Locality: sourcing logic, caching, install, and provenance all live behind one port
  • Fills the meta.json gap: provenance is auto-written at install, not a deferred manual step

Surface the Sync Flatten Semantics

scripts/sync-skills.sh basename dedup

Worth exploring in-process
Before
After

Files

  • scripts/sync-skills.sh:44-48

Problem

The sync script's realpath flattening is a seam with implicit collision semantics: when two skills share a basename, the last os.walk entry silently overwrites the first. This is a correctness bug hidden by convention (naming uniqueness hasn't been enforced).

Solution

Surface collision handling as an explicit step in the walk. Choose a strategy: pick-first (current behaviour but logged), skip-and-warn (safe default), or namespace-prefix. Make the strategy configurable and document it in the --help output.

  • Locality: collision logic lives in one place, not scattered across silent dest writes
  • Leverage: once resolved, the collision rule applies uniformly to all 153 synced skills
  • Prevents data loss: a silent overwrite today could destroy a skill; explicit handling surfaces it
  • Test surface: one test with two colliding basenames validates the behaviour
